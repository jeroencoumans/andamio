Dependencies

Some third-party micro-frameworks are included to support additional functionality, without the framework itself relying too much on it. For mobile websites and applications, these are:

  • Zepto, for DOM manipulation and AJAX requests. It's API is compatible with jQuery, so for desktop environments it can easily be replaced by jQuery
  • Fastclick for removing the 300ms delay when using click events
  • Zepto.scroll.js, a tiny plugin to allow manipulating the scroll position within elements (used to support statusbar tap to scroll to the top)
  • Swipeview, which allows the creation of infinite loopable carousels
  • For creating development and production distributions, Andamio depends on Grunt, which depends on nodejs:

For running the unit tests, CasperJS, and PhantomJS need to be installed.

There's a README.md included in the framework that explains these dependencies for development.

Design

There are two major components in Andamio: a set of basic CSS patterns to create the most common website/webapp elements, and a set of Javsacript modules that allow the developer to create basic interaction patterns.

CSS patterns:

The CSS patterns are coded in Less and can be found in the style/ directory. This directory contains main.less file which includes the major components. Each component contains a component.less file which includes the individual patterns. This makes it easy for the frontend-developer to include only the required patterns, override the theming of the patterns where needed, and include any custom patterns. For building project-specific CSS, it's trivial to adapt the default grunt.js.

Note that the CSS patterns are not in any dependent on the Javascript modules. The naming of a design pattern is the same for CSS, Javascript and unit test (e.g. loader.less & loader.js).

  • mixins: a set of mixins that normalize vendor-prefixes, which can be used by the frontend-developer to easily create custom styles
  • base: normalize styling of HTML elements, float-based grid, table-based grid, positioning and typography.
  • page: sets up the layout of the page, by default includes patterns for fixed headers, centered loader, slidable navigation, tabs and parent/child views
  • lists: bullet, definition and numbered lists, media object lists (newsfeed pattern), table-cell lists (iOS-style lists with arrows) and tables
  • ui: commonly used UI elements, like badges, buttons, pill-buttons, dividers, forms, icons, liquid images and video's, etc.
  • theme: contains the default theming for all patterns, like colors, gradients, shadows, typography etc.

Javascript modules:

The Andamio Javascript modules are in the js/app/ directory, the external libraries are in the js/lib/ directory. The following conventions are used:

The Javascript files follow the module pattern in the APP namespace. The structure of a module is as follows:

  • APP.module = (function () { }
  • First declare module-specific variables and constants
  • Then define private helper functions, if any
  • function attachListeners() includes all event listeners
  • function init() initialises variables, checks & sets the default state, and attaches listeners
  • return by default includes init, and any other public interfaces.
  • Interface names are kept as short, semantic and predictable as possible.
  • Modules can be called from your project Javascript or by using structured HTML with optional data- attributes.
  • Event listeners are always prefixed with the action- class - no ID's are used to make them reusable.
  • Structural page elements that should only be defined once (like the page navigation or the global loader) use the element's ID.
  • All modules are initialized from init.js, and init.js is initialized by calling APP.core.init() on document.ready.
  • Some modules depend on CSS styling (like modals or parent/child elements). In that case, both the functionality as well as the styling is tested in the function tests with CasperJS.
  • Javascript functionality is attached to DOM elements by using the js- prefix on the class. These classes should not include styling (the CSS linter checks for that).

Unit testing

Unit tests are created in CasperjJS, which runs in PhantomJS.

Functional and unit tests are found in the tests/ directory. The run_tests.sh script executes all unit tests in CasperJS. Tests are split up by module, and each test describes the interfaces, elements, actions and CSS that is tested. This is still work in progress, there are still some modules that have no tests.

Creating production & development builds

  • The default task lints and uglifies the Javascript files, and uses grunt-recess (which uses Twitter Recess) to lint, compile and minify the Less files.
  • There's a watch task that checks for changes in the JS and CSS files, and will concat those into main.dev.js and main.dev.css
  • The source files do not need to be present in the static directory of your project; only the distribution builds are required.

Documentation

The current documentation includes a description of the CSS classes that can be used as well as examples. The documentation for Javascript still needs to be done, although the modules themselves are heavily commented. A proper documentation site is a forthcoming hackaton project, until then, some documentation can be found here: jeroencoumans.github.com/andamio/

Besides documentation, a set of boilerplate HTML is available. This makes it easy for non-frontend developers to create a clickable prototype that can be styled and implemented in the project.
