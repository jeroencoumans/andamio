What is Andamio?

Andamio is a frontend framework to help you create a near-native mobile app experiences using only HTML, CSS and Javascript. It includes default interaction patterns and styles to quickly create high-fidelity prototypes that can be further developed for production builds. It assumes a mobile-first approach where you start with creating the interaction design for the mobile website, and enhance it for specific platforms using Phonegap. Since all interaction patterns are liquid, it's also very easy to extend it to support tablet and even desktop experiences.

These docs are meant to help you quickly setup a new project with Andamio. The preview window on the right shows you a sample of the interaction patterns, go ahead and try them!

Andamio's design philosophy

Andamio includes a lot of building blocks that take out the grunt work of setting up a website.

CSS patterns:

The CSS patterns are coded in Less and can be found in the style/ directory. This directory contains main.less file which includes the major components. Each component contains a .less file which includes the individual patterns. This makes it easy for the frontend-developer to include only the required patterns, override the theming of the patterns where needed, and include any custom patterns. For building project-specific CSS, it's trivial to adapt the default grunt.js.

Note that the CSS patterns are not in any dependent on the Javascript modules. The naming of a design pattern is the same for CSS, Javascript and unit test (e.g. loader.less & loader.js).

mixins:
a set of mixins that normalize vendor-prefixes, which can be used by the frontend-developer to easily create custom styles
base:
normalize styling of HTML elements, float-based grid, table-based grid, positioning and typography.
page:
sets up the layout of the page, by default includes patterns for fixed headers, centered loader, slidable navigation, tabs and parent/child views
lists:
bullet, definition and numbered lists, media object lists (newsfeed pattern), table-cell lists (iOS-style lists with arrows) and tables
ui:
commonly used UI elements, like badges, buttons, pill-buttons, dividers, forms, icons, liquid images and video's, etc.
theme:
contains the default theming for all patterns, like colors, gradients, shadows, typography etc.

Javascript modules:

The Andamio Javascript modules are in the js/app/ directory, the external libraries are in the js/lib/ directory. The following conventions are used:

  • The Javascript files follow the module pattern in the APP namespace. The structure of a module is as follows:
  • Interface names are kept as short, semantic and predictable as possible.
  • Modules can be called from your project Javascript or by using structured HTML with optional data- attributes.
  • Event listeners are always prefixed with the action- class - no ID's are used to make them reusable.
  • Structural page elements that should only be defined once (like the page navigation or the global loader) use the element's ID.
  • All modules are initialized from init.js, and init.js is initialized by calling APP.core.init() on document.ready.
  • Some modules depend on CSS styling (like modals or parent/child elements). In that case, both the functionality as well as the styling is tested in the function tests with CasperJS.
  • Javascript functionality is attached to DOM elements by using the js- prefix on the class. These classes should not include styling (the CSS linter checks for that).

Unit testing

Unit tests are created in CasperjJS, which runs in PhantomJS.

Functional and unit tests are found in the tests/ directory. The run_tests.sh script executes all unit tests in CasperJS. Tests are split up by module, and each test describes the interfaces, elements, actions and CSS that is tested. This is still work in progress, there are still some modules that have no tests.

Creating production & development builds

The default task lints and uglifies the Javascript files, and uses grunt-recess (which uses Twitter Recess) to lint, compile and minify the Less files.

There's a watch task that checks for changes in the JS and CSS files, and will concat those into main.dev.js and main.dev.css

The source files do not need to be present in the static directory of your project; only the distribution builds are required.
